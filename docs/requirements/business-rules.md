# 业务规则集 (Business Rules)

> **项目**: internal-user-service  
> **规则总数**: 18 条  
> **最后更新**: 2026-06-21

每条规则编号 `BR-NNN`，被代码 / 测试 / 文档引用。

---

## 身份规则

### BR-001: 邮箱唯一性
公司内部**每个邮箱地址只能对应一个有效用户**（不含已软删）。
- **冲突处理**: 创建时返回 409 Conflict
- **大小写**: 邮箱比较忽略大小写（存小写）
- **测试**: `tests/integration/test_user_create.py::test_duplicate_email`

### BR-002: 工号唯一性
员工工号（`employee_id`）在公司内全局唯一，且不可变更。
- **格式**: `E` + 8 位数字（如 `E01234567`）
- **冲突处理**: 创建时返回 409 Conflict

### BR-003: SSO 强制
所有内部员工**必须**通过公司 SSO 登录，禁止本地密码登录。
- **例外**: 系统账号（service account）使用 API key
- **执行**: API 层校验 `auth_method == 'sso'`

## 组织规则

### BR-004: 单主部门
每个员工**同时只能属于一个主部门**（`primary_department_id`）。
- **可兼职**: 可在多个 `secondary_departments`（最多 5 个）
- **变更**: 通过 HR 系统同步，本服务不直接修改

### BR-005: 部门层级深度限制
组织树深度不超过 **6 层**（公司 → 一级 → 二级 → 三级 → 小组 → 个人）。
- **越级处理**: 创建时返回 422 Validation Error
- **预留**: 预留 1 层用于未来"事业部"调整

### BR-006: 直属上级必须同部门
员工直属上级（`manager_id`）必须与员工属于同一部门。
- **例外**: CEO/CTO 等高管可跨部门
- **校验时机**: 创建 / 变更时

## 权限规则

### BR-007: 最小权限原则
新建用户默认仅授予 `employee` 角色，不预授权任何资源。
- **额外权限**: 必须由 Admin 显式授予

### BR-008: 权限继承
用户继承其**主部门及所有父部门**的角色。
- **不继承**: 兼职部门的角色不继承
- **生效**: 实时计算（不缓存）

### BR-009: 权限时效性
所有权限变更在 **30 秒内**对所有 API 调用生效。
- **实现**: 不缓存权限，DB 实时读取 + Redis 5 分钟 TTL

### BR-010: Admin 不能删除自己
用户不能删除自己或降低自己的 admin 角色（防误操作）。
- **执行**: 删除 / 改角色前校验
- **绕过**: 必须由另一个 admin 操作

## 数据规则

### BR-011: PII 字段加密存储
PII 字段（`phone`、`id_card`、`address`）在数据库中加密存储。
- **算法**: AES-256-GCM
- **密钥**: 从 Vault 读取，**永不进代码**

### BR-012: 用户删除采用软删
用户**不物理删除**，仅设置 `deleted_at`。
- **保留期**: 7 年（合规要求）
- **硬删**: 仅 GDPR 主动申请 + 法务审批后

### BR-013: 审计日志不可篡改
所有权限变更、登录失败、敏感操作必须写入审计日志。
- **写入位置**: Kafka `audit.events` topic
- **保留**: 7 年（合规要求）
- **完整性**: HMAC 签名 + 定期对账

## 生命周期规则

### BR-014: 离职即时失效
员工离职后，**所有 session 和 token 立即失效**。
- **实现**: 设置 `user.deactivated_at` 后广播事件，所有依赖服务 revoke
- **延迟**: ≤ 10 秒

### BR-015: 休眠账户清理
**180 天未登录**的员工自动进入休眠状态（`dormant=true`）。
- **清理**: 休眠后 30 天未恢复，禁用账号
- **例外**: 长期休假员工（HR 标记）

### BR-016: 密码复杂度
服务账号密码必须满足：
- 长度 ≥ 16 字符
- 包含大小写、数字、特殊字符
- 90 天强制更换

### BR-017: 邮箱变更需验证
员工变更邮箱必须经过**双因子验证**（旧邮箱 + 新邮箱各一次确认链接）。
- **链接有效期**: 24 小时
- **生效**: 两次确认后

### BR-018: API 调用配额
每个用户每分钟最多 **1000 次** API 调用（防滥用）。
- **超额**: 返回 429 + Retry-After
- **Admin**: 不受限

## 规则变更流程

1. PM 在 Slack `#user-service-product` 提 RFC
2. EM + Tech Lead + SRE Lead review（3 个 Approve）
3. 写入本文件，更新规则编号
4. 通知所有依赖方（Slack `#user-service-announce`）
5. 代码改动关联新规则编号

## 规则违反处理

| 违规等级 | 处理 |
|----------|------|
| 🔴 严重（数据损坏/安全） | 立即修复 + P0 事故 + 全员复盘 |
| 🟡 一般（体验降级） | 下个 Sprint 修复 |
| 🟢 轻微（边缘场景） | Backlog 排期 |
